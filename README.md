# 售后工单质量分析看板

面向客服主管的 **纯前端本地应用**，无需后端服务，所有数据存储于浏览器 localStorage。

支持导入客服工单 CSV、回访评分 CSV、退款 JSON，按配置规则自动识别 **超时工单 / 低分回访 / 重复投诉 / 高额退款** 四类质量问题；同一客户在时间窗口内的多条线索自动归并为"质量事件"，每条证据保留原始数据。支持人工复核、状态流转与全量数据导出/恢复。

## 完整复现步骤（覆盖所有核心需求）

### ▶️ Step 1：生成并导入样例数据

1. 启动项目后访问系统，左侧导航进入 **「数据导入」** 页面
2. 点击绿色大按钮 **「一键生成样例数据并导入」**
3. 观察页面反馈：
   - Toast 提示导入成功：`工单 X 条 + 评分 Y 条 + 退款 Z 条 + 质量事件 N 个`
   - 下方「导入历史」列表出现 3 条记录（工单 CSV / 评分 CSV / 退款 JSON），分别展示：
     - 总条数 / 有效条数 / 无效条数（部分样例行故意包含异常）
     - 导入时间、文件哈希末 8 位
4. ✅ **预期结果 A：数据校验保护生效**
   - 点开最新一次导入的「查看校验报告」，可见：
     - 工单：至少有 2 条无效（1 条 customer_id 为空、1 条 created_at 缺失）
     - 评分：至少有 2 条无效（1 条 customer_id 为空、1 条 score=99 非法）
     - 退款：至少有 1 条无效（1 条 amount=-100 负数）
   - **无效行被跳过，不影响已有数据完整性**

### ▶️ Step 2：验证退款文件重复导入保护

1. 仍在「数据导入」页面
2. 在「导入历史」中找到最新的退款记录，点击 **「下载原始样例文件」**（或直接在样例数据生成目录中找到 `sample_refunds_*.json`）
3. 把这个 JSON 文件再次拖入第三个上传区（退款 JSON 红色区域）
4. ✅ **预期结果 B：重复导入保护生效**
   - Toast 弹出红色错误：`该退款文件已导入过(记录ID: xxx)，拒绝重复导入`
   - 刷新页面或查看导入历史，**已有退款数据完全不变，数据不被破坏**

### ▶️ Step 3：配置规则并执行分析

1. 进入左侧导航 **「规则配置」** 页面
2. 确认默认规则（可按需修改）：
   | 规则 | 默认值 | 合法范围 | 非法示例 |
   |------|--------|----------|----------|
   | 超时阈值（小时） | 24 | 正整数 | 0、-5、3.5 |
   | 低分阈值（分） | 3 | 1-5 整数 | 0、6、2.5 |
   | 重复投诉窗口（天） | 7 | 1-365 整数 | -1、400 |
   | 重复投诉次数阈值 | 3 | ≥2 整数 | 1 |
   | 高额退款阈值（元） | 500 | ≥0 数字 | -100 |
3. ✅ **预期结果 C：规则配置合法性校验**
   - 尝试将「超时阈值」设为 `0` → 输入框立即红框，下方显示错误文字「必须是正整数」
   - 此时点击「保存配置」→ Toast 红色错误：`保存失败：timeout_hours - 必须是正整数`
   - **非法值不会写入 store，已有规则保持原值不变**
4. 将规则恢复为合法值后（或保留默认），点击 **「保存配置」** → Toast 提示保存并重新分析成功

### ▶️ Step 4：在看板查看质量事件与证据

1. 进入首页 **「分析看板」**
2. 顶部 5 张统计卡片：
   - **事件总数**（琥珀色）：样例数据通常产生 3-6 个事件
   - **待复核**（红色）：新生成事件默认状态为 pending
   - **复核中 / 已关闭**：尚未操作时应为 0
   - **涉及客户数**：C001、C002、C003 等高频投诉客户
3. 中间 4 张类型卡片（超时 / 低分 / 重复投诉 / 高额退款）：显示按类型命中的事件数
4. 下方事件表格：点击任意一行 → 右侧弹出 **「事件详情抽屉」**
5. ✅ **预期结果 D：事件归并与证据链**
   - 找到客户 `C001` 的事件，通常命中「重复投诉 + 超时 + 低分 + 高额退款」多种类型
   - 抽屉底部「证据时间线」按时间排序展示 4-8 条证据：
     - 🔵 工单证据：展开可看到原始 ticket_no、title、created_at、resolved_at
     - 🟣 评分证据：展示 1-2 分低分、具体评语
     - 🔴 退款证据：展示 899 / 1200 元高额退款金额、原因
   - **每条来源证据独立保留原始字段快照，可溯源**

### ▶️ Step 5：事件复核、状态流转与关闭

1. 进入 **「事件复核」** 页面
2. 左侧复核队列按状态排序，默认展示所有「待复核」和「复核中」事件
3. 选中队列中第一个待复核事件：
   - 在右侧「复核备注」输入：`已联系客户致歉，补发商品，预计3天送达`
   - 点击 **「开始复核」** 按钮 → 状态变为「复核中」（蓝色徽标）
   - 点击 **「关闭事件」** 按钮 → 弹窗确认「确认关闭该事件？关闭后无法重开」→ 确定
4. 再选第二个待复核事件，直接点击 **「批量选中」** 勾选，再点顶部 **「批量开始复核」**
5. ✅ **预期结果 E：状态与备注正确写入**
   - 顶部关闭时间自动记录（格式：YYYY-MM-DD HH:mm:ss）
   - 备注文字完整显示

### ▶️ Step 6：刷新验证持久化（最关键验证点）

1. **按 Ctrl+R / F5 刷新整个浏览器页面**
2. 重新进入系统后：
   - ✅ **预期结果 F-1 事件状态保持**：之前关闭的事件仍显示「已关闭」，复核中的事件仍为「复核中」
   - ✅ **预期结果 F-2 复核备注保持**：之前输入的 `已联系客户致歉...` 备注完整恢复
   - ✅ **预期结果 F-3 证据列表保持**：每个事件下的证据数量、顺序、原始数据完全一致
   - ✅ **预期结果 F-4 关闭时间保持**：`closed_at` 时间戳精确到秒，与关闭前一致
   - ✅ **预期结果 F-5 导入历史保持**：导入记录列表与刷新前完全相同（含文件哈希）
   - ✅ **预期结果 F-6 规则配置保持**：规则配置页的 5 个参数与刷新前一致

### ▶️ Step 7：导出 CSV / JSON / 全量备份

1. 进入 **「数据导出」** 页面
2. **按范围过滤导出：**
   - 勾选「状态筛选」→ 仅勾选「已关闭」
   - 勾选「类型筛选」→ 仅勾选「高额退款」
   - 勾选「包含证据明细」
   - 格式选 **CSV**
   - 观察预览统计：`当前选择将导出 X 条事件 / Y 条证据`
3. 点击三个导出按钮：
   - **「导出事件」** → 下载 `events_时间戳.csv`

   - **「导出全量备份」** → 下载 `full_backup_时间戳.json`（包含所有数据+规则+状态）
4. ✅ **预期结果 G：导出文件格式正确**

2. **破坏性测试（验证保护机制）：**

## 四类质量筛查规则逻辑说明

| 规则 | 判定条件 | 命中类型 |
|------|----------|----------|
| 超时工单 | `resolved_at - created_at > timeout_hours` | `timeout` |
| 低分回访 | `score < min_score`（score 必须为 1-5 整数） | `low_score` |
| 重复投诉 | 同一 `customer_id` 在 `repeat_days` 天内工单数量 ≥ `repeat_count`（滑动窗口） | `repeat_complaint` |
| 高额退款 | `amount > high_refund_amount` | `high_refund` |

### 事件归并逻辑

1. 所有命中的线索先转为 `Evidence`，保留 `raw_data` 快照
2. 按 `customer_id` 分组并按时间排序
3. 对每条证据：若存在同客户的未归档事件，且 证据时间 − 事件最后时间 ≤ `repeat_days × 2` 天，则加入该事件；否则创建新事件
4. 事件 `types` 取所有证据命中类型的并集，`first/last_seen_at` 取时间区间 min/max，`total_refund` 累加关联退款

## 字段校验规则

| 数据类型 | 必填字段 | 校验规则 | 失败处理 |
|----------|----------|----------|----------|
| 工单 CSV | `customer_id`, `ticket_no`, `created_at` | customer_id 非空；created_at 可解析（16种格式）；resolved_at 非空时也须可解析 | 记录行号+字段+原因后跳过该行 |
| 评分 CSV | `customer_id`, `score`, `visited_at` | score 必须为 1-5 整数；visited_at 可解析 | 同上 |
| 退款 JSON | `customer_id`, `refund_no`, `amount`, `refunded_at` | amount 非负数字；refunded_at 可解析 | 同上 |
| 退款文件级 | SHA-256 文件哈希 | 与历史 `importRecords` 对比，若存在相同哈希（且类型为 refund），**直接拒绝整个文件** | 不写入任何退款数据 |

## 数据持久化

- **存储位置**：浏览器 `localStorage`，键名 `quality-dashboard-state-v1`
- **持久化字段**（7 个核心集合 + 配置）：`tickets`, `scores`, `refunds`, `events`, `evidences`, `importRecords`, `rules`
- **不持久化**：`uiState`（当前选中事件、抽屉开关、最近分析时间等临时 UI 状态）
- **Date 处理**：序列化时 `Date` 自动转 ISO 字符串；反序列化时通过正则匹配并递归转回 `Date` 对象，**保证时间戳精度到毫秒级**
- **版本号**：v1（未来数据结构变更时可通过版本号做迁移）

## 安全与数据保护

| 场景 | 保护机制 |
|------|----------|
| 缺少 customer_id | 行级校验，记录错误原因并跳过，不写入 |
| 时间字段无法解析 | 16 种日期格式尝试，全部失败则标记错误行 |
| 规则配置非法（负数/超范围/非整数） | `validateAllRules` 统一检查，错误不写入，已有规则保留 |
| 重复导入同一退款文件 | 导入前 SHA-256 哈希比对，整个文件直接拒绝，不破坏已有退款 |
| 分析产生脏数据 | `runAnalysis` 每次先清空旧 events/evidences 再重新生成，幂等 |
| 重启丢失状态 | Zustand persist → localStorage，Date 字段精确恢复 |
| 导入错误隔离 | 每行独立 try-catch，单条失败不影响其他 99 条数据 |
