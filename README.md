# 售后工单质量分析看板

面向客服主管的 **纯前端本地应用**，无需后端服务，所有数据存储于浏览器 localStorage。

支持导入客服工单 CSV、回访评分 CSV、退款 JSON，按配置规则自动识别 **超时工单 / 低分回访 / 重复投诉 / 高额退款** 四类质量问题；同一客户在时间窗口内的多条线索自动归并为"质量事件"，每条证据保留原始数据。支持人工复核、状态流转与全量数据导出/恢复。

---

## 快速启动

```bash
# 1. 安装依赖
npm install

# 2. 启动开发服务器
npm run dev
# 默认访问 http://localhost:5173 (使用 HashRouter，file:// 直接打开 dist/index.html 也可)

# 3. 生产构建
npm run build
# 产物位于 dist/

# 4. 本地预览构建产物
npm run preview

# 5. TypeScript 类型检查
npm run check
```

## 完整复现步骤（覆盖所有核心需求）

### ▶️ Step 1：生成并导入样例数据

1. 启动项目后访问系统，左侧导航进入 **「数据导入」** 页面
2. 点击绿色大按钮 **「一键生成样例数据并导入」**
3. 观察页面反馈：
   - Toast 提示导入成功：`工单 X 条 + 评分 Y 条 + 退款 Z 条 + 质量事件 N 个`
   - 下方「导入历史」列表出现 3 条记录（工单 CSV / 评分 CSV / 退款 JSON），分别展示：
     - 总条数 / 有效条数 / 无效条数（部分样例行故意包含异常）
     - 导入时间、文件哈希末 8 位
4. ✅ **预期结果 A：数据校验保护生效**
   - 点开最新一次导入的「查看校验报告」，可见：
     - 工单：至少有 2 条无效（1 条 customer_id 为空、1 条 created_at 缺失）
     - 评分：至少有 2 条无效（1 条 customer_id 为空、1 条 score=99 非法）
     - 退款：至少有 1 条无效（1 条 amount=-100 负数）
   - **无效行被跳过，不影响已有数据完整性**

## 四类质量筛查规则逻辑说明

| 规则 | 判定条件 | 命中类型 |
|------|----------|----------|
| 超时工单 | `resolved_at - created_at > timeout_hours` | `timeout` |
| 低分回访 | `score < min_score`（score 必须为 1-5 整数） | `low_score` |
| 重复投诉 | 同一 `customer_id` 在 `repeat_days` 天内工单数量 ≥ `repeat_count`（滑动窗口） | `repeat_complaint` |
| 高额退款 | `amount > high_refund_amount` | `high_refund` |

### 事件归并逻辑

1. 所有命中的线索先转为 `Evidence`，保留 `raw_data` 快照
2. 按 `customer_id` 分组并按时间排序
3. 对每条证据：若存在同客户的未归档事件，且 证据时间 − 事件最后时间 ≤ `repeat_days × 2` 天，则加入该事件；否则创建新事件
4. 事件 `types` 取所有证据命中类型的并集，`first/last_seen_at` 取时间区间 min/max，`total_refund` 累加关联退款

---

## 字段校验规则

| 数据类型 | 必填字段 | 校验规则 | 失败处理 |
|----------|----------|----------|----------|
| 工单 CSV | `customer_id`, `ticket_no`, `created_at` | customer_id 非空；created_at 可解析（16种格式）；resolved_at 非空时也须可解析 | 记录行号+字段+原因后跳过该行 |
| 评分 CSV | `customer_id`, `score`, `visited_at` | score 必须为 1-5 整数；visited_at 可解析 | 同上 |
| 退款 JSON | `customer_id`, `refund_no`, `amount`, `refunded_at` | amount 非负数字；refunded_at 可解析 | 同上 |
| 退款文件级 | SHA-256 文件哈希 | 与历史 `importRecords` 对比，若存在相同哈希（且类型为 refund），**直接拒绝整个文件** | 不写入任何退款数据 |

## 数据持久化

- **存储位置**：浏览器 `localStorage`，键名 `quality-dashboard-state-v1`
- **持久化字段**（7 个核心集合 + 配置）：`tickets`, `scores`, `refunds`, `events`, `evidences`, `importRecords`, `rules`
- **不持久化**：`uiState`（当前选中事件、抽屉开关、最近分析时间等临时 UI 状态）
- **Date 处理**：序列化时 `Date` 自动转 ISO 字符串；反序列化时通过正则匹配并递归转回 `Date` 对象，**保证时间戳精度到毫秒级**
- **版本号**：v1（未来数据结构变更时可通过版本号做迁移）

## 安全与数据保护

| 场景 | 保护机制 |
|------|----------|
| 缺少 customer_id | 行级校验，记录错误原因并跳过，不写入 |
| 时间字段无法解析 | 16 种日期格式尝试，全部失败则标记错误行 |
| 规则配置非法（负数/超范围/非整数） | `validateAllRules` 统一检查，错误不写入，已有规则保留 |
| 重复导入同一退款文件 | 导入前 SHA-256 哈希比对，整个文件直接拒绝，不破坏已有退款 |
| 分析产生脏数据 | `runAnalysis` 每次先清空旧 events/evidences 再重新生成，幂等 |
| 重启丢失状态 | Zustand persist → localStorage，Date 字段精确恢复 |
| 导入错误隔离 | 每行独立 try-catch，单条失败不影响其他 99 条数据 |
| 复盘包事件快照丢失 | 创建时深拷贝所有事件字段到快照，原事件修改/删除不影响历史数据 |
| 复盘包导入冲突 | 同时检查 ID 和标题冲突，冲突包跳过并明确提示，不静默覆盖 |
| 复盘包操作追溯 | 所有关键操作（创建/追加备注/状态变更/导入/删除）均记录审计日志 |
